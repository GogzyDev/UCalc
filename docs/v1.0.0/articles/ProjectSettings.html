Project Settings

UCalc exposes several configuration options that control how expressions are parsed, compiled, and how symbols are discovered.

These settings can be found in:

Project Settings → UCalc

Contexts

These settings control how symbol contexts are built and what members are discovered.

Context Population Mode

Controls how type contexts populate their members.

Lazy

Members are discovered only when first used during compilation.

Benefits:

  • lower memory usage
  • faster initialization

This is the default option and generally recommended for most projects.

Eager

All usable members are discovered immediately when the context is created.

Benefits:

  • slightly faster expression compilation
  • fewer runtime lookups

Drawback:

  • higher memory usage.

Note:
This setting only affects runtime builds.
The Unity Editor always uses Eager mode to support autocomplete in expression fields.

Include Non-Public Static Members

When enabled, static member discovery will include:

  • private
  • protected
  • internal

members.

By default only public static members are exposed.

Include Non-Public Instance Members

When enabled, instance symbol discovery will include:

  • private
  • protected
  • internal

members from the expression owner.

This can be useful when expressions need to access internal implementation details.

Tokens

These settings control how expression tokens are compared.

Token Comparison

Defines how identifiers are compared during parsing and symbol lookup.

Example modes include:

  • Ordinal
  • OrdinalIgnoreCase

For example, with OrdinalIgnoreCase:

health  
Health  
HEALTH

would all resolve to the same symbol.

This setting also affects:

  • argument names
  • dependency names
  • global symbol providers
  • custom providers

Performance

These options control internal optimizations used by UCalc.

Compile Parlot Parsers

When enabled, Parlot parsers are compiled to improve parsing performance.

This can significantly reduce the cost of repeatedly parsing expressions.

Disabling this option may slightly reduce startup cost but generally results in slower parsing.

Compile Using Fast Expressions

When enabled, UCalc uses FastExpressionCompiler to compile expression trees.

Benefits:

  • faster delegate compilation
  • improved runtime performance

If disabled, UCalc will fall back to the standard:

Linq.Expression.Compile()

method.
